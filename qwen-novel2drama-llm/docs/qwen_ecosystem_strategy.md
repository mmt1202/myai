# Qwen 生态扩展策略：从“小说转短剧”到“AI 短剧全链路工厂”

`qwen-novel2drama-llm` 第一版聚焦“小说文本 → 短剧结构化资产”，这是最小可训练闭环，不代表产品边界只能停留在小说转短剧。

更合理的长期定位是：

```text
AI 短剧全链路模型工厂
= 文本编剧 + 视觉理解 + 分镜导演 + 视频提示词 + 配音台词 + ASR/TTS + Agent 工作流
```

## Qwen 生态模块映射

| Qwen 家族方向 | 在 AI 短剧平台中的作用 | 本项目阶段 |
|---|---|---|
| Qwen Text / Instruct | 小说理解、大纲、分集剧情、角色设定、分镜、提示词 | P0 已支持 |
| Qwen3 / 后续文本模型 | 更强推理、复杂剧情、多轮创作、工具调用 | P1 规划 |
| Qwen-VL / Qwen3-VL | 参考图理解、角色图一致性检查、分镜图审核 | P2 规划 |
| Qwen-TTS | 配音生成、角色声音设计、旁白生成 | P2 规划 |
| Qwen-ASR | 用户语音输入、配音稿回听转写、素材字幕 | P2 规划 |
| Qwen-Omni | 图文音视频统一理解与生成工作流 | P3 研究 |
| Qwen-Agent | 自动化短剧生产流程、工具编排、任务队列 | P3 规划 |
| Qwen-Coder | 平台代码生成、插件生成、工作流工具开发 | 工程辅助 |

## 为什么第一版仍然先做小说转短剧？

1. 训练成本最低：文本 SFT 最容易形成闭环。
2. 数据最容易标注：小说片段、分镜、角色、提示词都能人工质检。
3. 业务价值明确：短剧制作链路的上游资产都来自剧本结构。
4. 可扩展性强：后续 VL/TTS/ASR/Omni 都可以围绕文本资产继续扩展。

## 产品边界升级建议

第一版名称可以继续保留 `qwen-novel2drama-llm`，但产品路线建议升级为：

```text
qwen-drama-factory
```

模块拆分：

```text
qwen-drama-writer      文本编剧模型
qwen-drama-storyboard  分镜导演模型
qwen-drama-vision      角色/画面一致性模型
qwen-drama-voice       配音/TTS/ASR 模块
qwen-drama-agent       自动化生产 Agent
qwen-drama-api         统一业务 API
```

## 当前工程如何兼容未来扩展？

- `configs/`：继续放文本 LoRA/QLoRA 配置。
- `prompts/`：扩展 VL/TTS/ASR/Agent 提示词。
- `inference/`：增加多模型路由，而不是只加载一个文本模型。
- `eval/`：增加图像一致性、语音质量、字幕准确率等评估模板。
- `docs/`：沉淀各模型家族选型和接入方式。

## 下一步建议

1. 保持 P0：先把文本短剧编剧模型训练出来。
2. P1：加入 Qwen3 文本模型配置和多轮创作数据。
3. P2：新增 Qwen-VL 角色图/分镜图理解脚手架。
4. P2：新增 TTS/ASR 配音与字幕工作流文档。
5. P3：用 Qwen-Agent 编排“小说 → 剧本 → 分镜 → 视频提示词 → 配音稿”的自动化流水线。
